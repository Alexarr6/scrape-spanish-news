### Important repo truth
- story clustering is currently **not embedding-driven**
- same-story grouping is built by `ClusterPipeline.build_clusters()` over enriched articles
- the real architecture today is already: pair scoring -> accepted edges -> guarded graph closure -> persisted clusters
- Explorer semantic clustering (HDBSCAN over embeddings) is a different subsystem

### Most likely under-grouping causes noted
1. cluster input slice is recency-limited and not candidate-retrieval aware
2. heuristic key phrases are too weak for same-event matching across outlets
3. title similarity is superficial (`SequenceMatcher`)
4. closure logic is deliberately conservative and can split real follow-ups / pivot-linked stories
5. threshold defaults differ between manual CLI and stories wrapper
6. there is no evaluation dataset / metric loop for same-event quality

### Planning decisions recorded
- do **not** recommend replacing story clustering with semantic HDBSCAN
- do recommend reframing as same-event matching pipeline:
  - candidate generation
  - pair scorer v2
  - graph closure v2
- do recommend keeping embeddings/pgvector as auxiliary recall signals, not the sole clustering engine
- do recommend separating near-duplicate vs same-event conceptually, at least in metrics/rules first

### External research note
Short external pass on cross-document event coreference supports the general direction that event matching benefits from entity/time/action-aware linking and clustering, but the repo is not ready for a heavy learned coreference system yet. The immediate fit remains a stronger retrieval+pairwise+graph pipeline.

### What changed
- introduced `_generate_candidate_pairs(...)` inside `ClusterPipeline`
- candidate stage now unions four bounded origins:
  - `temporal_window`
  - `shared_tag`
  - `shared_entity`
  - `lexical_neighbor`
- pair artifacts now include:
  - `candidate_origins`
  - `candidate_rank`
- rebuild metrics now include:
  - `candidate_origin_counts`
  - `candidate_overflow_counts`
- evaluation scaffold now includes `candidate_recall_summary` with recall@k based on candidate rank

### Design choices worth preserving
- kept cost bounded with per-origin and per-seed caps
- kept temporal gate hard (`max_days_delta`)
- did not replace pair scoring; candidate generation only decides what gets scored
- made origin tracing first-class so future passes can see whether misses are due to retrieval or scoring
